# Design System: Internationalization (i18n)

This document covers OSCER's internationalization conventions, locale file organization, naming patterns, and the workflow for translating Figma text into i18n keys. OSCER supports English (`en`) and US Spanish (`es-US`) locales.

> **Machine-readable version**: `.claude/rules/design-system-i18n.md` contains the compact rule set consumed by AI tooling. This document is the expanded human-readable reference.

## Foundational Rules

1. **Always use `t()` for ALL user-visible text.** Never hardcode English strings in ERB templates, controllers, mailers, or helpers.
2. **Both locales must have translations.** Every key added to `en.yml` must also appear in the corresponding `es-US.yml` file (even as a placeholder during development).
3. **Use lazy lookup `t(".key")` in views.** Rails auto-scopes these to the current controller and action path.
4. **Use full-path lookup `t("namespace.key")` outside views.** This includes helpers, controllers, mailers, services, and Ruby code.

### Organizational principles

- **One view locale file per controller**: `config/locales/views/{controller_name}/en.yml`. Keys inside match the action/partial names.
- **Model locales are separate**: `config/locales/models/{model_name}/en.yml` holds `activerecord.models` and `activerecord.attributes` keys.
- **Defaults hold shared UI**: Header text, flash messages, date formats, and form builder defaults live in `config/locales/defaults/en.yml`.
- **Services hold backend text**: Mailer content, notification messages, and status reason codes live in `config/locales/services/en.yml`.
- **Dashboard uses sub-files**: State-specific dashboard card content (e.g., `activity_report_approved.yml`) is split into separate files for maintainability.

## Key Naming Conventions

### View keys

View keys mirror the controller/action/partial hierarchy. Rails lazy lookup (`t(".key")`) resolves based on the current template path.

```yaml
# config/locales/views/activities/en.yml
en:
  activities:                          # controller name
    new:                               # action name
      title: "Activity type"
      back: "Back to activity report"
      continue: "Continue"
    form:                              # partial: _form.html.erb
      title_prefix: "Add"
      name: "Organization name"
      continue: "Save and continue"
    show:                              # action: show
      title:
        work: "Work for %{name}"
      back: "Back to activity report"
      edit: "Edit activity"
      destroy: "Delete activity"
```

### Model keys

Model keys use the standard Rails `activerecord` namespace:

```yaml
# config/locales/models/activity/en.yml
en:
  activities:
    evidence_sources:
      self_reported: "Self Reported"
      ai_assisted: "AI Assisted"
      ai_assisted_with_member_edits: "AI Assisted with Edits"
      ai_rejected_member_override: "AI Rejected"
    confidence:
      low_label: "Low confidence"
```

### Shared/default keys

```yaml
# config/locales/defaults/en.yml
en:
  date:
    formats:
      default: "%m/%d/%Y"
      short: "%b %d"
      long: "%B %d, %Y"
  helpers:
    submit:
      create: "Submit"
      update: "Update"
  us_form_with:
    boolean_true: "Yes"
    boolean_false: "No"
    optional: "Optional"
```

### Application-wide view keys

```yaml
# config/locales/views/application/en.yml
en:
  header:
    title: "Medicaid Work Reporting Portal"
    account: My account
    dashboard: Dashboard
    logout: "Log out"
  flash:
    error_heading:
      zero: "An unexpected error occurred"
      one: "An error occurred"
      other: "%{count} errors occurred"
  breadcrumbs:
    home: "Home"
```

## Lookup Patterns

### Lazy lookup (views and partials)

In ERB templates, `t(".key")` auto-resolves to the current view path:

```erb
<%# In app/views/activities/new.html.erb %>
<%# t(".title") resolves to en.activities.new.title %>
<h1><%= t(".title") %></h1>

<%# In app/views/activities/_form.html.erb %>
<%# t(".name") resolves to en.activities.form.name %>
<%= f.text_field :name, label: t(".name") %>
```

### Full-path lookup (controllers, helpers, services)

Outside of views, always use the full key path:

```ruby
# In a controller
flash[:notice] = t("exemption_screener.success.application_created")

# In a helper
def status_label(status)
  t("member_statuses.#{status}")
end

# In a mailer
subject = t("services.member_status_service.reason_codes.age_under_19_exempt")
```

### Scoped lookup

When you need to reference keys from a different view within a template:

```erb
<%# Reference a key from the application namespace %>
<%= t("header.title") %>

<%# Reference a key from another controller's view %>
<%= t("dashboard.index.greeting", first_name: @user.first_name) %>
```

## Common Code Patterns

### Page title

```erb
<% content_for :title, t(".title") %>
```

### Form labels and hints

```erb
<%= f.text_field :name, label: t(".name"), hint: t(".name_hint") %>
```

### Button text

```erb
<%= f.submit t(".save") %>
<%= f.submit t(".continue"), big: true %>
<%= link_to t(".back"), parent_path, class: "usa-button usa-button--outline" %>
```

### Interpolation

Use `%{variable}` placeholders in YAML and pass the values as keyword arguments:

```yaml
# en.yml
en:
  dashboard:
    index:
      greeting: "Hi %{first_name}"
  activities:
    show:
      title:
        work: "Work for %{name}"
  exemption_screener:
    may_qualify:
      exemption_type: "Exemption type: %{exemption_name}"
```

```erb
<%= t("dashboard.index.greeting", first_name: @user.first_name) %>
<%= t(".title.work", name: @activity.name) %>
<%= t(".exemption_type", exemption_name: @exemption.name) %>
```

### Flash messages

```ruby
# Controller
flash[:notice] = t(".success_message")
flash[:errors] = t(".error_message")
```

### Alert components

```erb
<%= render AlertComponent.new(
  type: :error,
  heading: t("flash.error_heading", count: @errors.count)
) do |c| %>
  <% c.with_body do %>
    <ul class="usa-list">
      <% @errors.each do |error| %>
        <li><%= error.full_message %></li>
      <% end %>
    </ul>
  <% end %>
<% end %>
```

## Pluralization

Rails uses the `count` parameter to select the correct plural form:

```yaml
# config/locales/views/application/en.yml
en:
  flash:
    error_heading:
      zero: "An unexpected error occurred"
      one: "An error occurred"
      other: "%{count} errors occurred"
```

```erb
<%= t("flash.error_heading", count: @errors.count) %>
<%# count=0 -> "An unexpected error occurred" %>
<%# count=1 -> "An error occurred" %>
<%# count=5 -> "5 errors occurred" %>
```

For English, provide `one` and `other` keys (and optionally `zero`). For Spanish (`es-US`), provide the same structure -- Rails handles the grammatical rules per locale.

## Date and Time Formatting

OSCER defines custom date/time formats in `config/locales/defaults/en.yml`:

```yaml
en:
  date:
    formats:
      default: "%m/%d/%Y"       # 03/26/2026
      short: "%b %d"            # Mar 26
      long: "%B %d, %Y"        # March 26, 2026
  time:
    formats:
      default: "%B %d, %Y"     # March 26, 2026
      long: "%B %d, %Y %-I:%M %p"  # March 26, 2026 2:30 PM
```

Usage in views:

```erb
<%# Date formatting with l() (localize) helper %>
<%= l(date, format: :default) %>      <%# 03/26/2026 %>
<%= l(date, format: :short) %>        <%# Mar 26 %>
<%= l(date, format: :long) %>         <%# March 26, 2026 %>

<%# Time formatting %>
<%= l(time, format: :long) %>         <%# March 26, 2026 2:30 PM %>

<%# With timezone helper %>
<%= local_time(time, format: :long) %>
```

The USWDS date picker format hint is also localized:

```yaml
en:
  us_form_with:
    date_picker_format: "Format: mm/dd/yyyy"
```

## YAML Gotchas

### Boolean-like values must be quoted

YAML interprets `true`, `false`, `yes`, `no`, `on`, `off` as booleans. Always quote these when they appear as string values:

```yaml
# WRONG - YAML interprets these as booleans
en:
  buttons:
    yes_answer: yes         # parsed as boolean true
    no_answer: no           # parsed as boolean false

# CORRECT - quoted strings
en:
  buttons:
    yes_answer: "Yes"
    no_answer: "No"
```

The `en.yml` files in this project include a reminder comment at the top:

```yaml
# Be aware that YAML interprets the following case-insensitive strings as
# booleans: `true`, `false`, `on`, `off`, `yes`, `no`. Therefore, these strings
# must be quoted to be interpreted as strings.
```

### Interpolation placeholder syntax

Always use `%{variable}` (percent-brace) syntax, not `#{variable}` (Ruby string interpolation):

```yaml
# CORRECT
en:
  greeting: "Hi %{first_name}"

# WRONG - Ruby interpolation doesn't work in YAML
en:
  greeting: "Hi #{first_name}"
```

### Indentation

YAML is whitespace-sensitive. Use exactly 2 spaces per indentation level. Do not use tabs.

### Key hierarchy must match the view path

For lazy lookup to work, the YAML key hierarchy must exactly match the controller/action path:

```
Controller: ActivitiesController
Action: new
Template: app/views/activities/new.html.erb
Partial: app/views/activities/_form.html.erb

YAML keys:
  en.activities.new.title        -> t(".title") in new.html.erb
  en.activities.form.name        -> t(".name") in _form.html.erb
```

Note that partials drop the leading underscore: `_form.html.erb` maps to the key `form`, not `_form`.

### Nested values under a key with a direct value

A YAML key cannot simultaneously hold a scalar value and child keys. If you need both, restructure:

```yaml
# WRONG - "title" can't be both a string and a parent
en:
  activities:
    show:
      title: "Activity Details"
      title:
        work: "Work for %{name}"

# CORRECT - use the base key for child variants only
en:
  activities:
    show:
      title:
        work: "Work for %{name}"
```

## Figma Text to i18n Keys

When translating a Figma design to code, follow this workflow for every text element:

### Step 1: Identify the text

Look at every text layer in the Figma component. Each piece of user-visible text (headings, labels, buttons, descriptions, hints, error messages) needs an i18n key.

### Step 2: Determine the locale file

Based on which controller will render the view, locate (or create) the correct locale file:

```
config/locales/views/{controller_name}/en.yml
```

### Step 3: Create descriptive keys

Use the lazy lookup convention with descriptive, snake_case key names:

```erb
<%# Figma text: "Tell us about your situation" (heading) %>
<h1><%= t(".heading") %></h1>

<%# Figma text: "Start" (button) %>
<%= f.submit t(".buttons.start") %>

<%# Figma text: "Some people don't need to report..." (description paragraph) %>
<p><%= t(".description") %></p>
```

### Step 4: Add to both locale files

```yaml
# config/locales/views/exemption_screener/en.yml
en:
  exemption_screener:
    index:
      heading: "Tell us about your situation"
      description: "Some people don't need to report their work hours. Answer a few questions to see if you are eligible for an exemption."
      buttons:
        start: "Start"
```

```yaml
# config/locales/views/exemption_screener/es-US.yml
es-US:
  exemption_screener:
    index:
      heading: "Cuéntenos sobre su situación"
      description: "Algunas personas no necesitan reportar sus horas de trabajo..."
      buttons:
        start: "Comenzar"
```

### Step 5: Verify -- no hardcoded strings

After implementation, search the ERB template for any remaining English text that is not wrapped in `t()`. The only exceptions are:
- HTML structural elements (`<div>`, `<section>`)
- CSS class names
- Data attributes
- Ruby method calls and variable references

Rails automatically applies a `translation_missing` CSS class (red dashed border) to any missing i18n keys in development mode, making gaps easy to spot visually.

## Adding a New Locale File

When adding a view for a new controller, create the locale file with this structure:

1. Create the directory if it does not exist:
   ```
   config/locales/views/{controller_name}/
   ```

2. Create `en.yml`:
   ```yaml
   en:
     {controller_name}:
       {action_name}:
         title: "Page Title"
         heading: "Page Heading"
   ```

3. Create `es-US.yml` with the same structure:
   ```yaml
   es-US:
     {controller_name}:
       {action_name}:
         title: "Título de la Página"
         heading: "Encabezado de la Página"
   ```

4. Rails auto-loads all `*.yml` files under `config/locales/` -- no manual registration is needed.

You can also use `make locale` from the `reporting-app/` directory to scaffold locale files for a model.
